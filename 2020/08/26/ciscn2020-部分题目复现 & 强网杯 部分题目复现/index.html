easybox

程序保护:

[*] ‘/home/user/easybox’
Arch: amd64-64-little
RELRO: Full RELRO
Stack: Canary found
NX: NX enabled
PIE: PIE enabled

漏洞点:

在add功能的函数中size很明显的加了一，存在off by one 漏洞

解题思路

构造chunk overlap, 劫持_IO_2_1_stdout结构体，泄露libc地址，时候再通过fast bin attack来打malloc_hook最后getshell

具体构造chunk overlap的过程:

add(0, 0xf8, ‘\n’)
add(1, 0x18, ‘\n’)
add(2, 0x68, ‘\n’)
add(3, 0xf8, ‘\n’)
add(4, 0x18, ‘\n’)
首先创建这么多chunk

free(0)

free(2)
add(2, 0x68, ‘\x00’ * 0x60 + p64(0x190) + ‘\x00’)

free(3)
先让chunk0进入unsorted bin，再通过chunk2来踩chunk3的标志位为0， 最后free chunk3触发合并

add(0, 0xf8, ‘\n’)
add(1, 0x18, ‘\n’)
add(5, 0x68, ‘\n’)
add(6, 0xf8, p64(0x21) * 2 * 0xf)
这时chunk5 和 chunk2 的地址是一样的

再是劫持stdout结构体的过程

free(2)
free(1) # change size
add(1, 0x18, ‘\x00’ * 0x18 + ‘\x91’)
free(5) # stdout
add(5, 0x88, ‘\xdd\x25’)
free(1)
add(1, 0x18, ‘\x00’ * 0x18 + ‘\x71’)
先让chunk2 进入fast bin，之后利用chunk1修改chunk2的size为0x91，再让chunk5进入unsorted bin，这样chunk2在fast bin中的fd，bk就是libc地址，之后回收再unsorted bin中的chunk，并且修改fd的低二位字节为stdout结构体中的fake chunk的偏移，之后再通过chunk1把size修改回0x71，之后就可以劫持stdout结构体了

'\x00'*0x33 + p64(0xFBAD1800) + p64(0)*3 + '\x88'

nofree

本题学习的是这位师傅的文章
CISCN 2020 初赛Pwn

程序保护:

[\*] '/home/user/nofree' Arch: amd64-64-little RELRO: Partial RELRO Stack: Canary found NX: NX enabled PIE: No PIE (0x400000)

漏洞点
strdup函数的原理是根据字符串的长度malloc一个大小合适的chunk，并返回这个chunk的地址，因此如果输入的size大于字符串的长度，是可以造成堆溢出的，根据规律，strcup生成chunk的size的规则是如果malloc(len(str) + 8)，没看源码，调试得到的结果是这样的

解题思路
由于本题没有free函数，所以需要利用这个堆溢出来实现free的效果，也就是house of orange

思路这位师傅的文章写的很清楚了，我简单的写下加深印象

  1. 由于存在堆溢出，可以修改top_chunk的size，并且程序中也可以无限申请chunk，那么可以考虑在bss段申请chunk，只要控制了在bss段的指针数组，就可以有任意地址写，大体方法是先修改top_chunk的size，然后不断申请chunk，在top_chunk的size为0x71的时候，只要申请一个size比0x70大的chunk，top_chunk就会进入fast bin中，结合堆溢出就可以控制fd指针，利用fast bin attack可以控制在bss段的指针数组

  2. 由于不知道靶机的libc版本，所以选择通过syscall来getshell，要进行syscall调用的话得控制ret，可以考虑用函数尾部的ret，这样的话需要泄露栈地址，知道栈地址后就可以利用任意地址写控制ret进行rop， 又因为elf文件中的通用gadget无法控制rax， 但是可以利用read函数的返回值来控制rax，所以也要泄露read函数的地址，得到read函数的地址和栈地址后就是往栈上布置rop，最后getshell

babymessage

程序保护

[*] '/home/user/babymessage' Arch: amd64-64-little RELRO: Partial RELRO Stack: No canary found NX: NX enabled PIE: No PIE (0x3fe000)

漏洞点:



在leave_message函数中在写v3变量时存在溢出，不过溢出长度最多溢出到rbp，无法控制ret

解题思路

  1. 经过观察可以知道如果leave_message函数中保存的rbp是main函数的rbp，当main函数结尾是再次执行leave ret，是可以造成栈迁移的，不过本题的溢出长度不够，栈迁移没有意义，在栈上还存留着start的地址，可以计算偏移修改rbp的低位字节使其指向start-8，这样在main函数结尾时执行ret时rsp就指向start，可以让程序重新执行，这样mm的大小就会增大，也就是溢出空间会增大
    需要说的是距离start的偏移不是固定的，可能需要多次运行才会成功

  2. 溢出空间增大后，可以跳转到 v1=100 处，有这么大的溢出空间后就方便进行rop泄露libc地址，调用system来getshell了
